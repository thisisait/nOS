### Faze 2: Pentest

| ID | Severity | Title | Confidence |
|----|----------|-------|------------|
| PENTEST-003 | HIGH | Code interpreter module blocking bypass via _real_import reference leak | high |
| PENTEST-004 | MEDIUM | Multi-hop prompt injection via tool result re-injection (30-retry loop) | medium |

**PENTEST-003 details**: The code interpreter's module blocking mechanism (middleware.py:2957-2977) prepends a wrapper that saves `builtins.__import__` as `_real_import`, then replaces it with `restricted_import`. The `_real_import` variable remains accessible in the LLM code execution scope. LLM-generated code (triggered by prompt injection) can reassign `builtins.__import__ = _real_import` to bypass all module restrictions. Default `CODE_INTERPRETER_BLOCKED_MODULES` is empty. When `CODE_INTERPRETER_ENGINE=jupyter` (server-side), this enables full RCE. Default engine is `pyodide` (client-side WASM), which mitigates the server-side impact.

**Positive mitigations found**: Tool name whitelist, parameter name filtering, `ast.literal_eval` (not `eval`), admin-gated tool creation, user-explicit tool selection per chat.
