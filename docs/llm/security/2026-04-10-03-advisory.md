- `src/db/models/organization.rs:38-118` — Membership model + MembershipType enum with access level ordering

- `src/api/core/organizations.rs:468-677` — Collection create/update/bulk-access with per-collection manage checks
- `src/api/core/organizations.rs:2697-2735` — Group member management
- `src/api/core/organizations.rs:2815-2873` — Password reset with role hierarchy
- `src/api/admin.rs:185-247` — Admin panel auth (constant-time comparison + rate limiting)
- `src/db/models/collection.rs:516-662` — Collection access SQL join chain + UPSERT logic

### Analysis

Vaultwarden 1.35.4 organization RBAC is well-implemented:

1. **Role hierarchy** is correctly enforced at the endpoint level (AdminHeaders, ManagerHeaders, OwnerHeaders)
2. **Cross-role operations** (promote/demote, invite, confirm non-Users) require the Owner role
3. **Collection management** validates per-collection manage permissions through a SQL JOIN chain
4. **Admin token** uses constant-time comparison (ct_eq / Argon2) with rate limiting
5. **Password reset** respects the role hierarchy (Owner > Admin > Manager/User)
6. **UPSERT logic** in CollectionUser::save prevents Manager self-promotion on new collections

**Minor observations** (not vulnerabilities):
- `put_group_members` does not validate the org membership (DB pollution only, no access bypass)
- An Admin can reset another Admin's password (by design per the Bitwarden model)
